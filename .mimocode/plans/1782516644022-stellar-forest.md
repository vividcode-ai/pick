# Plan: Fix Ctrl+Enter Bugs on Windows

## Problem
Two symptoms when pressing Ctrl+Enter on Windows terminals:
1. Text in first line is cleared (submitted as a message)
2. Continuing to type inserts stray newlines

## Root Cause Analysis

The Ctrl+Enter detection has a **timing vulnerability** on Windows. The flow is:

1. `process_key_event` (key_events.rs:95) receives `KeyCode::Enter`
2. `resolve_modifiers()` (key_events.rs:32) polls `GetAsyncKeyState(VK_CONTROL)` to detect Ctrl
3. If GetAsyncKeyState misses the Control key (user released Ctrl slightly before Enter event is processed), resolved modifiers = NONE
4. Falls through to `handle_key(Enter, NONE)` → `submit_input()` → **text cleared**

The 20ms `finalize_paste_accumulator` timer in the event loop can also interfere — it may flush accumulator text at a moment that creates timing conflicts with the Enter event processing.

## Key Files
- `crates/cli/src/modes/tui/key_events.rs` — `process_key_event`, `drain_key_events`, `resolve_modifiers`
- `crates/tui/src/app/handlers.rs:819` — `handle_key` (Ctrl+Enter arm at line 928)
- `crates/tui/src/editor/mod.rs:185` — `insert_newline_auto_indent`

## Fix Strategy

### Change 1: Make process_key_event more robust for Enter detection

In `process_key_event` (key_events.rs), when Enter arrives with no modifiers AND the paste accumulator is non-empty, **always** treat it as a newline insertion (not a submit). Currently lines 106-118:

```rust
// Current: only newline-izes if accumulator non-empty
if !tui.paste_accumulator.is_empty() {
    tui.paste_accumulator.push('\n');
    tui.last_paste_time = Some(now);
    tui.force_flush_paste_accumulator();
    return None;
}
// Current: falls through to submit
tui.force_flush_paste_accumulator();
return tui.handle_key(key.code, key.modifiers);
```

The issue: if user typed text that was already flushed from accumulator to editor (by the 20ms timer), the accumulator is empty when Enter arrives, so it falls through to submit even though the user intended Ctrl+Enter.

**Fix**: Also check `GetAsyncKeyState` as a secondary path BEFORE calling `handle_key` for submit. This is already done at line 96-104, but if `resolve_modifiers` fails (timing), we need a fallback.

Add a **debounce buffer** for Enter events: when Enter arrives with no modifiers, delay the submit by checking if a Ctrl-up event arrives within a short window, OR re-poll GetAsyncKeyState once more.

### Change 2: Add a second GetAsyncKeyState poll as fallback

Before the final `return tui.handle_key(key.code, key.modifiers)` at line 118, add one more `resolve_modifiers` check with a fresh timestamp:

```rust
// Last-chance check: re-poll OS key state
let recheck = resolve_modifiers(&key);
if recheck.intersects(KeyModifiers::SHIFT | KeyModifiers::CONTROL) {
    tui.force_flush_paste_accumulator();
    tui.editor.insert_newline_auto_indent();
    return None;
}
```

### Change 3: Same fix for drain_key_events

Apply identical debounce logic in `drain_key_events` (line 282-291) where bare Enter events during paste accumulation are handled. Currently bare Enter in drain is silently skipped, but if it arrives with CONTROL modifier from a second event, it could cause double newline insertion.

### Change 4: Same fix for process_key_event_during_agent

Mirror changes from Change 1 into `process_key_event_during_agent` (line 186-200).
